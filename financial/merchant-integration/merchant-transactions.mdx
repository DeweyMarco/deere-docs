## Overview

The Merchant Transactions API facilitates transaction processing and data integration for John Deere Financial merchants. It enables merchant Business Systems partnered with John Deere Financial to submit transactions for the following finance products:

- Multi-Use Account (US)
- Multi-Use Account (CAN)
- PowerPlan
- John Deere Financial Revolving Plan (JDFRP)

<Note>
  This API is intended for entities that provide Business Systems, ERP Systems, and/or CRM Systems connected with dealers and retailers who are registered as merchants offering one or more John Deere Financial revolving products.
</Note>

## Authentication

The Merchant Transactions API supports two OAuth 2.0 flows depending on your integration.

### Three-Legged OAuth 2.0 (Authorization Code)

Use this flow when acting on behalf of a user.

<Steps>
  <Step title="Create an application on Developer.Deere.com">
    Your application will be assigned a **Client Key** and **Client Secret**. Add one or more Callback URLs (Redirect URIs) in the OAuth Profile section of your application. If callback URLs fail to sync, contact support — a 400 Bad Redirect error will occur during auth.
  </Step>
  <Step title="Discover OAuth endpoints">
    ```bash
    GET https://signin.johndeere.com/oauth2/aus78tnlaysMraFhC1t7/.well-known/oauth-authorization-server
    ```
    Returns the authorization endpoint, token endpoint, and supported scopes.
  </Step>
  <Step title="Acquire an authorization code">
    Redirect the user to the authorization server with `response_type=code`, your `client_id`, `redirect_uri` (URL-encoded), `state`, and required scopes.

    | Scope | Description |
    |---|---|
    | `offline_access` | Request a refresh token |

    The user signs in with John Deere credentials and accepts the scope allowance screen. The server redirects back to your callback URL with the authorization code.
  </Step>
  <Step title="Exchange for an access token">
    POST to the token endpoint with `grant_type=authorization_code`, the authorization code, and redirect URI. The access token expires after **12 hours**.
  </Step>
  <Step title="Call the API">
    ```bash
    Authorization: Bearer <access_token>
    ```
  </Step>
  <Step title="Refresh the access token">
    Use `grant_type=refresh_token` before expiry. The refresh token expires after **365 days** of inactivity — if it expires, the user must re-authenticate from step 3.
  </Step>
</Steps>

### Client Credentials (Machine-to-Machine)

Use this flow for server-to-server integrations that don't require user context.

```bash
POST https://signin.johndeere.com/oauth2/aus78tnlaysMraFhC1t7/v1/token
```

<CodeGroup>

```bash cURL
curl --request POST \
  --url 'https://signin.johndeere.com/oauth2/aus78tnlaysMraFhC1t7/v1/token' \
  --header 'Authorization: Basic <base64(clientId:clientSecret)>' \
  --header 'Accept: application/json' \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data 'grant_type=client_credentials&scope={customScope}'
```

</CodeGroup>

<Note>
  Pass the client ID and secret in the HTTP `Authorization` header using HTTP Basic Auth — not in the request body.
</Note>

```json
{
  "token_type": "Bearer",
  "expires_in": 43200,
  "access_token": "eyJhbG[...]1LQ",
  "scope": "customScope"
}
```

Tokens expire after **12 hours**. Refresh as needed.

## Transaction Flows

### Authorization → Capture (two-step purchase)

Use this when you need to reserve funds at order time and capture later (e.g., on shipment).

```
POST /merchants/transactions/authorizations   → reserve credit
GET  /merchants/transactions/authorizations/{id}  → check status
POST /merchants/transactions/captures         → finalize on delivery
```

### Direct Purchase (single-step)

Use this for immediate point-of-sale transactions.

```
POST /merchants/transactions/purchases
```

### Returns & Corrections

```
POST   /merchants/transactions/credits        → issue a credit
DELETE /merchants/transactions/credits/{id}   → void if not yet settled
```

### Customer Payments at Merchant Location

```
POST   /merchants/transactions/payments       → record payment
DELETE /merchants/transactions/payments/{id}  → void if not yet settled
```

### Captures

#### Capture Approved Authorization

Consume credit previously reserved on a credit line. Typically triggered when a purchase is finalized or product is shipped/delivered. A capture of an approved authorization is effectively a purchase.

```
POST /merchants/transactions/captures
```

<ParamField body="authorizationId" type="string" required>
  ID of the approved authorization to capture.
</ParamField>

<ParamField body="amount" type="number" required>
  Amount to capture. Must not exceed the authorized amount.
</ParamField>

<ParamField body="invoiceNumber" type="string">
  Merchant invoice number associated with this capture.
</ParamField>
