## Overview

A **boundary** is a **multipolygon** that outlines a **field**. It has one or more **exterior** rings and may include **interior** rings for areas that should not be worked (waterways, ditches, obstacles).

In John Deere **Operations Center**, field boundaries are **optional**: a field may have **zero**, **one**, or **many** boundaries. When several exist, one is often marked **active**. A field without boundaries can still contain other data. If your product requires a boundary, you can **generate** one from a **Field Operation** and persist it with **POST**.

<CardGroup cols={2}>
  <Card title="List & read" icon="list">
    **GET** `/platform/organizations/{orgId}/boundaries` and **GET** `/platform/organizations/{orgId}/fields/{fieldId}/boundaries` — discover boundaries at org or field scope; **GET** by id for a single resource.
  </Card>
  <Card title="Create & update" icon="pencil">
    **POST** and **PUT** require **`ag3`**. Updates must include **name**, **active**, **irrigated**, **sourceType**, and **multipolygons**; activating one boundary deactivates others on the same field.
  </Card>
  <Card title="Generate from operation" icon="map">

## Getting started (missing boundaries)

If every field must have at least one boundary:

1. **Find fields without boundaries** — use the Fields **list** endpoint with `?embed=boundaries` (or equivalent); if **`boundaries`** is empty, none exist.
2. **List field operations** — **GET** `/platform/organizations/{orgId}/fields/{fieldId}/fieldOperations` and let the user pick an operation (e.g. from map previews) or apply your own selection rules.
3. **Preview** — **GET** `/platform/fieldOperations/{operationId}/boundary` to retrieve a generated boundary; adjust **name** or interior rings as needed.
4. **Persist** — **POST** `/platform/organizations/{orgId}/fields/{fieldId}/boundaries` to store the boundary in Operations Center.

## Boundary object (summary)

A **boundary** is a polygon or set of polygons. It must have at least one **exterior** ring; additional exteriors and/or **interiors** are allowed. In list responses, each item may appear under a **`boundaries`** collection on the field.

| Name | Type | Description |
| --- | --- | --- |
| `name` | string | Display name (often the field name). |
| `area` | measurement (double) | Total area (`value` + `unit`). |
| `multipolygon` | multipolygon | One or more polygons. |
| `extent` | extent | Bounding rectangle for all points. |
| `active` | boolean | Whether this is the active boundary for the field (default **false**). |
| `signalType` | string | Signal used when defining the boundary; see **dtSignalType** in the representation system. |

### Complex types

**Area (`measurement` AsDouble)**

| Name | Type | Description |
| --- | --- | --- |
| `value` | double | Numeric amount. |
| `unit` | string | Unit of measure. |

**Polygon**

| Name | Type | Description |
| --- | --- | --- |
| `rings` | ring[] | List of rings; first ring is an **exterior**. |

**Ring**

| Name | Type | Description |
| --- | --- | --- |
| `points` | point[] | Vertices; first and last should match (minimum closed shape is a triangle). |
| `type` | string | e.g. **`exterior`** (field border) or **`interior`** (blocked area). |
| `passable` | boolean | Default **true**. |

### Ring types (conceptual)

| Type | Meaning |
| --- | --- |
| **exterior** | Outer border of the field. |
| **interior** | Blocked area inside the field. |

### dtSignalType (examples)

`dtiSignalTypeUnknown`, `dtiSignalTypeDigitized`, `dtiSignalTypeNoDiff`, `dtiSignalTypeUnknownDiff`, `dtiSignalTypeWAAS`, `dtiSignalTypeSF1`, `dtiSignalTypeSF2`, `dtiSignalTypeRTKX`, `dtiSignalTypeRTK`, `dtiSignalTypeSF3`, `dtiSignalTypeSFRTK`, `dtiSignalTypeNonJdDiff`, `dtiSignalTypeJdDiff`

## Boundary events (Data Subscription Service)

Boundary **events** fire on create, update, delete, archive/unarchive, and related field merge/unmerge or field archive changes. Subscriptions require an **`orgId`** filter; **`fieldId`**, **`boundaryId`**, **`active`**, **`status`**, and **`action`** filters are optional.

| Status | Description |
| --- | --- |
| **ARCHIVED** | Boundary is archived. |
| **AVAILABLE** | Boundary is available (not archived). |

| Action | Description |
| --- | --- |
| **CREATED** | Boundary was created. |
| **DELETED** | Boundary was deleted. |
| **MODIFIED** | Metadata update, archive/unarchive, field merge/unmerge, or field archive/unarchive affecting the boundary. |

Prefer **one organization per subscription**: you cannot change **`orgId`** list after creation, and the subscription ends if the user loses access to any listed org.

## Authentication (OAuth 2.0)

John Deere **Precision Tech** APIs use **OAuth 2.0** with the **authorization code** grant. Create an application on [Developer.Deere.com](https://developer.deere.com/) for your **Application Id** and **Secret**. Register **Redirect URIs**; mismatched URIs can return **400 Bad Redirect**.

<Steps>
  <Step title="Discover endpoints and scopes">
    ```bash
    GET https://signin.johndeere.com/oauth2/aus78tnlaysMraFhC1t7/.well-known/oauth-authorization-server
    ```
  </Step>
  <Step title="Authorize">
    **GET** the authorize URL with `response_type=code`, scopes (**`ag1`**, **`ag2`**, and/or **`ag3`** as required), **client_id**, **state**, and a **URL-encoded** `redirect_uri` registered on your application.
  </Step>
  <Step title="Exchange the code for tokens">
    **POST** `https://signin.johndeere.com/oauth2/aus78tnlaysMraFhC1t7/v1/token` with `grant_type=authorization_code`. Access tokens expire after **12 hours**. Request **`offline_access`** for a **refresh token**.
  </Step>
  <Step title="Enable organization access">
    Call **GET** `/platform/organizations`. If a link has **`rel=connections`**, redirect the user so they can grant access. When complete, **`manage_connections`** appears instead.
  </Step>
  <Step title="Call Boundaries APIs">
    Send **`Authorization: Bearer {token}`** and **`Accept: application/vnd.deere.axiom.v3+json`**. For **POST**/**PUT**, send **`Content-Type: application/vnd.deere.axiom.v3+json`**.
  </Step>
</Steps>

### OAuth scopes for these endpoints

| Scope | Description |
| --- | --- |
| `ag1` | View locations — **GET** list/read boundaries |
| `ag2` | Analyze production data — **GET** generate boundary from field operation |
| `ag3` | Manage locations and production data — **POST**, **PUT**, **DELETE** boundaries |

Request only scopes your application needs. User permissions cap what the connection allows.


The body must include **name**, **active**, **irrigated**, **sourceType**, and **multipolygons**. Setting **active** to **true** marks other boundaries on the field inactive. A license may be required for **sourceType** values other than **External**.

<ParamField path="fieldId" type="string" required>

</ParamField>

## Additional notes

- **Field list with boundaries:** Use the Fields API with an **embed** parameter (for example `embed=boundaries`) to see whether **`boundaries`** is populated per field.
- **403 / terms:** Some orgs return **403** with **X-Deere-Terms-Location** when user action is required (see platform **RCA** guidance).
